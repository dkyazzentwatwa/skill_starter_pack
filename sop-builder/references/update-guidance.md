# SOP Update Guidance

Use this guidance when the user asks how to update an SOP or when the workflow appears to be drifting.

## What should trigger an SOP update

Update the SOP when any of these change:
- the tool stack or system of record changes
- ownership or approval steps change
- the trigger or intake path changes
- new exceptions appear repeatedly
- QA checks, review thresholds, or success criteria change
- the output format changes
- escalations are going to a different team or role
- operators are consistently improvising around missing instructions

## What to review first

Review these sections first because they drift fastest:
1. Trigger
2. Required Inputs / Tools
3. Procedure
4. Quality Checks
5. Exceptions / Edge Cases
6. Escalation Path
7. Revision Notes

## How to update the SOP cleanly

1. Identify the operational change.
2. Locate every section affected by that change.
3. Rewrite the procedure, not just the summary.
4. Update QA checks so they match the new workflow.
5. Update exceptions and escalation logic if the change affects routing or approvals.
6. Add a revision note with date, owner, and change summary.
7. If the change is large, reissue the checklist companion too.

## When to recommend a full revalidation

Recommend a stronger review when:
- the workflow is external-facing or customer-facing
- money, compliance, account access, or legal review is involved
- multiple teams hand off work across systems
- repeated failures show that the SOP no longer matches reality
- the old SOP was built from incomplete notes and is assumption-heavy

## Revision note pattern

Use this pattern unless the user has a house format:
- Last updated: [date]
- Updated by: [role or name]
- Change summary: [what changed]
- Reason: [why it changed]
- Follow-up needed: [optional]
